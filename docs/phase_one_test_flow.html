一眼看流程

输入文件

selected file + raw text

从 data/01.txt 读出纯文本，写入 raw_input.txt 和 selected_file.json。

测试脚本第 1179-1194 行。

Case Structurer

RawTextInput + StageContext

原文被包成带 case_id、input_id、received_at 的输入对象，同时判断这是初始输入。

结构化

Sections + Items

先切成临床段落，再从段落里抽出 StructuredClinicalItem。这一步还不拆成证据原子。

校正

corrected_result

规范 section/item id，解析 source span，并做 source span validation correction。

Evidence Tree Structurer

TreeStructuringCandidate

把 item、section、source span 合并成每个 item 一个候选对象。

这是内部对象，不落盘。

额外 LLM 层

ClinicalObjectAssertion

从候选 item 中再抽“临床对象 + 肯否定状态 + 范围”，例如把咳嗽、咳痰、胸闷、气短拆开。

树形结构

EvidenceTree

每个 item 生成一个 tree，tree 里是带父子关系的 tree_nodes。

输出 + 校验

JSON + HTML report

写出 clinical_object_assertions.json、evidence_trees.json、item_to_tree_links.json 和可读 report.html。

Schema 和 Validator 怎么插在流程里

这里分两类：schema 是 Pydantic 数据形状约束；validator/corrector 是 pipeline 里显式调用的业务一致性检查或校正。

执行位置 主要 schema 显式 validator / guard / corrector 检查什么 结果怎么传
Case Structurer: RawInputBuilder RawTextInput Pydantic model validation raw_text 非空；生成并约束 input_id、case_id、input_order、parent_input_id。 传给 StageContextExtractor、ClinicalSectionExtractor。
Case Structurer: stage / section / item extraction StageContext、ClinicalSection、StructuredClinicalItem、SourceSpan Pydantic model validation 字段枚举值、必填字段、source span input_id 一致性、item 必须有 source span。 进入 normalizer 和分阶段 source span validation/correction。
Case Structurer: normalize / resolve ClinicalSection、StructuredClinicalItem SectionNormalizer、SectionSourceSpanValidator、SectionSourceSpanCorrector、ItemNormalizer、ItemSourceSpanValidator、ItemSourceSpanCorrector 规范 id 和顺序；先验证/修补 section spans，再验证/修补 item spans。 传给 assembler。
Case Structurer: assemble CaseStructuringResult Pydantic cross-object validation stage/input 的 case_id、input_id 匹配；section/item 属于同一 input；section/item id 唯一。 生成 corrected result。
Case Structurer: staged correction SourceSpanValidationReport、SourceSpanCorrectionReport、SectionSourceSpanValidationCorrectionResult、ItemSourceSpanValidationCorrectionResult validate_and_correct_section_spans、validate_and_correct_item_spans Section 输出后检查 section provenance；item 输出后检查 item provenance。能确定修复的就修复。 测试脚本取 corrected_result，直接传给 Evidence Tree Structurer。
Evidence Tree Structurer: preflight TreeStructuringWarning EvidenceTreeStructurerInputGuard 检查 case_id/input_id/source_structuring_result_id 是否一致；上游 ready flag；是否有 structured_items；item 是否都有 source_spans。 有 ERROR 就直接返回空 EvidenceTreeStructuringResult，不跑后续 LLM。
Evidence Tree Structurer: candidate build TreeStructuringCandidate Pydantic model validation 把 item、section、source spans 合成 LLM-ready candidate；保证内部字段形状稳定。 传给 assertion resolver 和 tree builder。
Evidence Tree Structurer: assertion resolve ClinicalObjectAssertion、ClinicalAssertionResolutionResult ClinicalAssertionValidator LLM 输出的 object_text 必须是 candidate.source_text 连续子串；source_item_id 不匹配会修正；object_type/status/confidence 非法会修正；cue/scope 必须能落回原文。 assertions 传给 tree builder；同时落盘为 clinical_object_assertions.json。
Evidence Tree Structurer: tree build EvidenceTree、EvidenceTreeNode、EvidenceTreeBuildResult EvidenceTreeValidator tree_nodes 必须是数组；node_text 必须是 source_text 连续子串；parent/context/assertion/span 引用必须存在；node_origin 必须和 provenance 一致；非法 node 会丢弃或修正。 合法 trees 直接作为最终科研检查结构；同时落盘为 evidence_trees.json。
Evidence Tree Structurer: result assemble ItemEvidenceTreeLink、EvidenceTreeStructuringResult、EvidenceTreeStructuringValidationReport Pydantic validation、引用校验 组装完整 evidence tree、item-to-tree links 和 warnings；不再生成额外证据对象。 输出 evidence_tree_structuring_result.json；有 tree 且没有 ERROR warning 时 ready_for_hypothesis_state=true。

用 item_002 看一条数据怎么流

1. 原文片段变成 item

主因“间断咳嗽咳痰伴胸闷气短8年，加重2月”入院

Case Structurer 生成：

item_002，item_type=symptom，temporality=chronic，source_span=span_item_002

2. item 内找属性

attribute_...8c995263：span_text=8年，role=symptom_duration，normalized_text=8 years

attribute_...cd12cd8c：span_text=加重2月，role=worsening_interval，normalized_text=2 months

3. item + attribute 合成 candidate

TreeStructuringCandidate.item_id=item_002，里面带着 item 基本字段、source spans、section 信息和上面两个 attributes。

4. candidate 再拆 assertions

生成多个 ClinicalObjectAssertion：间断咳嗽、咳痰、胸闷、气短、加重。

5. assertions 变成树

tree 直接表达主诉/入院事件、症状对象、属性和修饰关系；每个 node 通过 node_origin 与 provenance 说明它来自 assertion、上下文还是结构性分组。

6. 当前到这里停止

科研检查优先看 tree node、parent、relation、node_origin、source_assertion_ids/source_span_ids 和 warnings；下游直接消费完整树。

已删除的复杂度

旧版本的复杂尾链已经从代码里删除。 当前 Evidence Tree Structurer 的科研链路变成：

Item -> TreeStructuringCandidate -> ClinicalObjectAssertion -> EvidenceTree

  1. ClinicalObjectAssertion 是对象级拆分层：它先决定一个 item 里有哪些临床对象，以及对象是 present 还是 absent。
  2. EvidenceTree 是树形结构层：它把对象组织成父子节点，并保留 assertion/span provenance。
  3. 已删除：覆盖单元、额外证据对象 draft LLM 调用、normalizer、dedup、coverage validator。
  4. 后续调试优先看 assertion 是否合理、tree 是否合理，以及 node_origin/provenance 是否正确。
